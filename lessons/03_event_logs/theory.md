# Theory — Event Logs and Data Formats

## 1. What Is an Event Log?

Every behavioural experiment generates a **event log**: a timestamped record of discrete, datable occurrences during a session. An event log answers the question *what happened, and exactly when?* Common events include stimulus onset (the moment a visual stimulus appears on screen), participant response (a keypress or button press), feedback delivery, and experimenter annotations.

Franciscus Donders' 1868 reaction-time study is the earliest formalised example of event-log reasoning. His data table recorded, for each trial, the stimulus type and the elapsed time before the response — exactly the two columns that appear in every modern RT log. The conceptual structure has not changed in 150 years; only the precision and volume have grown.

Modern experiments can log thousands of events per second. An EEG recording at 1 000 Hz produces 1 000 sample-timestamps per second per electrode; a 64-channel cap yields 64 000 time-tagged values per second. Behavioural paradigms running in PsychoPy or E-Prime emit discrete trial-level events, but the underlying timing infrastructure can resolve sub-millisecond intervals when hardware is configured correctly.

## 2. Common Data Formats

### CSV and TSV

**Comma-Separated Values (CSV)** and **Tab-Separated Values (TSV)** are the universal workhorses of behavioural science data storage. Each row is one observation; each column is one variable; values are delimited by a comma or tab character.

Strengths: human-readable, editable in any spreadsheet application, parseable by every statistical software package (R, Python, SPSS, MATLAB). No special libraries required.

Weaknesses: flat structure only — nested or hierarchical metadata (e.g., electrode coordinates, channel names, session-level annotations) cannot be represented elegantly without additional sidecar files. Large files (>10 million rows) begin to strain memory.

### JSON

**JavaScript Object Notation (JSON)** is a text-based format that supports nested key-value structures, arrays, and mixed data types within a single document.

Strengths: hierarchical — a single JSON document can embed a participant's demographic information, session parameters, and all trial-level events in a naturally nested structure. Parseable by every modern programming language.

Weaknesses: verbose (keys are repeated for every record in a flat list), slower to parse than binary formats for large files, not directly importable into traditional statistical software without preprocessing.

### HDF5

**Hierarchical Data Format version 5 (HDF5)** is a binary format designed for large numerical datasets. It organises data into a filesystem-like hierarchy of groups and datasets, supports compression, and allows partial reads (reading a slice of a large array without loading the whole file).

Strengths: extremely efficient for large continuous signals (EEG, fMRI BOLD timeseries, accelerometer streams). MNE-Python, the dominant EEG/MEG analysis library, uses HDF5 internally. Files can be terabytes without performance degradation.

Weaknesses: not human-readable; requires dedicated libraries (h5py in Python, rhdf5 in R); version compatibility issues exist between HDF5 library versions.

### EDF and BDF

**European Data Format (EDF)** and its 24-bit extension **BDF** are the standard exchange formats for clinical and research EEG. EDF was designed in 1992 by Kemp et al. to transfer polysomnographic data between hospital systems. BDF extends EDF to 24-bit samples, giving it sufficient dynamic range for high-density EEG amplifiers.

Strengths: universally supported by EEG analysis software (EEGLAB, MNE-Python, BrainVision Analyzer, MATLAB). Used in clinical EEG worldwide, making EDF files immediately interpretable by any neurologist with standard equipment.

Weaknesses: fixed header structure makes it difficult to embed complex experimental metadata. Not suitable for non-EEG data.

### Format Trade-Off Summary

| Format | Human-readable | Hierarchical | Large data | Standard in field |
|---|---|---|---|---|
| CSV/TSV | Yes | No | Poor | Behavioural science |
| JSON | Yes | Yes | Poor | Web APIs, metadata sidecars |
| HDF5 | No | Yes | Excellent | Neuroimaging, EEG |
| EDF/BDF | No | No | Good | Clinical EEG |

## 3. Anatomy of a Behavioural Event Log

A well-structured behavioural log contains columns that together uniquely identify each observation and record everything needed for analysis. The following table describes the standard columns found in files generated by this application:

| Column | Type | Purpose |
|---|---|---|
| `participant_id` | string | Uniquely identifies the participant across all sessions and studies |
| `session_id` | string / integer | Identifies a single continuous recording session |
| `trial_id` | integer | Sequential trial number within the session; enables order-effect analysis |
| `condition` | string | Experimental condition label (e.g., `congruent`, `incongruent`, `go`, `nogo`) |
| `stimulus_onset_ms` | integer | Milliseconds since session start when the stimulus appeared on screen |
| `response_time_ms` | integer | Milliseconds from stimulus onset to participant response (RT) |
| `response_key` | string | The key or button the participant pressed |
| `correct` | boolean | Whether the response matched the expected response for that trial |

Every column serves an analytical purpose. `participant_id` and `session_id` enable multi-level modelling. `trial_id` enables sequential and practice-effect analyses. `condition` is the independent variable. `stimulus_onset_ms` enables time-alignment with physiological signals. `response_time_ms` is the dependent variable of primary interest. `response_key` enables error analysis. `correct` is required for accuracy-based exclusion criteria.

A log missing any of these columns forces the analyst to make assumptions — assumptions that are invisible and unverifiable by anyone reading the published paper.

## 4. The Time-Alignment Problem

Synchronisation of timestamps across devices is one of the most technically demanding problems in cognitive neuroscience. Consider a typical EEG experiment:

1. A **stimulus computer** runs PsychoPy and records the time of each stimulus onset using the operating system clock.
2. An **EEG amplifier** records brain activity continuously, timestamped by its own internal crystal oscillator.
3. A **response box** may use a third clock.

If these three clocks are not synchronised, the experimenter cannot align the brain signal to the stimulus. An unsynchronised 20 ms drift between the stimulus computer and the EEG amplifier would corrupt all event-related potential (ERP) averages, because the "N200" component centred at 200 ms post-stimulus would actually be a smeared average of signals occurring at 180–220 ms.

### TTL Triggers

The standard solution is **TTL (Transistor-Transistor Logic) triggers**: the stimulus computer sends a brief electrical pulse through a parallel port or trigger cable simultaneously with each stimulus onset. The EEG amplifier records this pulse as a special channel alongside the brain data, timestamped by the amplifier's own clock. Because the trigger arrives in the EEG data stream at the same instant as the stimulus, trial onset times can be reconstructed exactly in the amplifier's timebase — regardless of any difference between the two clocks.

### Software-Only Latency

Experiments running without hardware triggers (e.g., online studies in jsPsych, or behavioural-only tasks without physiological recording) face software-level timing uncertainty. Standard operating system keyboard polling occurs at 8–15 ms intervals on most consumer hardware, meaning that the logged keypress timestamp may lag the actual keypress by up to 15 ms. This floor must be acknowledged when interpreting small RT differences: a 10 ms effect is within measurement noise on consumer hardware.

High-precision behavioural timing requires either dedicated hardware response boxes (accurate to 1 ms or better) or display-synchronised software frameworks that latch stimulus onset to a vertical screen refresh.

## 5. Sampling Rates and Temporal Resolution

The **sampling rate** of a recording system determines the finest temporal difference it can resolve. The Nyquist theorem states that a system sampling at frequency *f* can faithfully represent signals up to *f/2*. For event-level data, the implication is:

- **EEG at 1 000 Hz**: one sample every 1 ms. Temporal resolution sufficient to separate the P100 (100 ms) from the N200 (200 ms) visual ERP components.
- **Keyboard polling at 125 Hz** (standard USB HID): one sample every 8 ms. Temporal resolution sufficient for typical RT comparisons (effect sizes of 50+ ms), but not for sub-10 ms effects.
- **Eye-tracking at 250–1 000 Hz**: one sample every 1–4 ms. Sufficient for saccade onset detection, which requires ~1 ms precision.

Understanding the precision floor of your recording system prevents over-interpretation of small numerical differences. If two conditions differ by 12 ms in mean RT but the response device has 15 ms timing jitter, the difference is uninterpretable.

## 6. Open Data Standards: BIDS

The **Brain Imaging Data Structure (BIDS)** is a community standard for organising and describing neuroimaging and electrophysiology datasets (Gorgolewski et al., 2016, *Scientific Data*). BIDS specifies:

- A directory layout (e.g., `sub-01/ses-01/eeg/sub-01_ses-01_task-flanker_eeg.bdf`)
- Required and optional JSON sidecar files describing acquisition parameters
- A standardised events file format (TSV with columns `onset`, `duration`, `trial_type`)
- A dataset-level `dataset_description.json` and participant-level `participants.tsv`

The core motivation for BIDS is **self-description**: a properly formatted BIDS dataset contains enough metadata that any researcher — even one unfamiliar with the original experiment — can understand what was recorded, how it was acquired, and what each file contains, without contacting the original authors.

**OpenNeuro** (openneuro.org) hosts thousands of BIDS-formatted datasets contributed by research groups worldwide. All datasets are freely downloadable. The platform runs automated validation (the BIDS Validator) to ensure compliance before public release.

The replication crisis in psychology and cognitive neuroscience was partly driven by the absence of data-format standards: different laboratories stored the same type of data in incompatible formats with different column names, units, and conventions, making independent reanalysis effectively impossible. BIDS is a structural response to that problem.

## 7. Historical Context: From Tables to Terabytes

Donders' 1868 data fit in a hand-written table. A contemporary EEG study with 40 participants, 64 channels, and 1 000 Hz sampling produces approximately 9 GB of raw data. A large-scale neuroimaging study (ABCD Study, n = 11 800) produces petabytes.

This growth in data volume created the need for standardised formats: when data is small, idiosyncratic storage formats are merely inconvenient. When data is large and shared across institutions, a missing column name or an ambiguous unit convention can waste weeks of analyst time and, in the worst case, produce incorrect scientific conclusions.

PhysioNet (physionet.org) hosts clinical physiological datasets — ECG, EEG, respiration — in open formats with extensive documentation, enabling researchers worldwide to develop and validate analysis algorithms on real human data without conducting new experiments.
